# 红绿灯审计 · W15 复核冻结输入（2026-07-08，规则 v1.1）

这是「红绿灯拦得住吗」那三个拦截数字（① 拦截率 76% / ② 误拦率 8% / ③ 🟢队列密度 86%）的
**冻结复现输入**。冻结的意思是：这里的文件不再改动，任何人 clone 仓库后能拿它独立重算出
台账 W15 与主 README 里的三个数，验证它们不是手搓的。生产池每天生长、词典还会迭代，那些
会漂移；这个目录钉死在 W15 那一刻。

学的是 `data/runs/2026-07-07-official-v1.2/`（官方指标冻结基线）的路子：把复现所需的输入
装箱冻结，跟每天变的生产数据解耦。

## 目录内容

| 文件 | 是什么 |
|---|---|
| `audit80_verdicts_raw.csv` | 80 个盲投候选（YouTube top50 + B站 top30）的**人工判级**：✅ 可直接联系 / ⚠️ 需人工核验 / ❌ 盲投会尴尬。每行 = 序号,频道名,粉丝数,判级,理由。已知正例名字带 " 🟡正例" 装饰后缀。 |
| `ranked_youtube_daily.json` | 当日 YouTube 全池排序（2,430 行），每行含 `action_grade` = 引擎**规则 v1.1** 红绿灯（🟢/🟡/🔴/🏢）。 |
| `ranked_bilibili.json` | 当日 B站全池排序（319 行），同样含 `action_grade`。 |
| `recompute_redlight.py` | 自足复算脚本：只读上面三份文件，纯标准库，不联网、不碰生产池、不 import 引擎源码，跑出 76%/8%/86%。 |

## 怎么复算

    cd data/runs/2026-07-08-audit-v1.1
    python3 recompute_redlight.py

预期输出：

    候选总数 80，join 命中 80，未命中 0
    ① 拦截率 = 25/33 = 75.8%   (README: 76%)
    ② 误拦率 = 2/25 = 8.0%   (README: 8%)
    ③ 🟢密度 = 18/21 = 85.7%  (README: 86%)

## 三数口径（与主 README 一致）

- **① 拦截率 25/33 ≈ 76%**：人工判 ❌ 的 33 个里，被引擎硬拦（🔴 或 🏢）25 个。剩下 8 个没有一个漏成 🟢——1 个（SAWAAR）是年级精度刻意释放，其余 7 个是平台（B站为主）无上传日期落进 🟡 安全网。
- **② 误拦率 2/25 = 8%**：人工判 ✅ 的 25 个里 2 个被引擎判 🔴。查下来是命中了已合作名单（`already_partner`）：引擎比盲判官多掌握了「它俩已经合作过」这条事实，把它们从新线索拨回存量维护，不是把好人拦错。按最严口径仍如实计为误拦。
- **③ 🟢密度 18/21 ≈ 86%**：引擎判 🟢 的 21 个里 18 个人工也判 ✅。

## 复现边界（诚实交代）

- 名字对齐用**精确匹配**（剥掉 CSV 的 " 🟡正例" 装饰后 80/80 全命中，无模糊匹配兜底）。
- `ranked_*.json` 的 `action_grade` 是**当日流水线产物**，用当时仓库里的 `config/insta360.json`（规则 v1.1）跑出。词典后续再收紧（如 R2-10 官号词表精化）会让个别 grade 变动，但那属于新一轮复核，本目录钉死在 W15。
- 拦截率的四舍五入：25/33 = 75.76%，README 记 76%；18/21 = 85.71%，记 86%。
